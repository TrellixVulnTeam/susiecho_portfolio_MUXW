Overview

프로젝트 요약

K-town은 베트남과 미국 동부에 거주하는 한인을 위한 로컬 비즈니스 검색 서비스이자 마케팅 플랫폼이다. 일반 사용자는 원하는 지역의 한인 업소 정보와 프로모션 혜택을 받을 수 있고, 다른 사용자들의 리뷰를 통해 추가적인 정보를 얻을 수 있다. 누구나 업소 등록이 가능하며 비용 지급을 통해 홍보를 진행할 수 있다.

나의 업무

앱 서비스에 대한 사용자들의 불만과 서비스 이탈률이 증가함에 따라, 사용자가 겪는 문제점을 파악하여 앱 서비스를 개선한다. 한정된 일정과 인력의 규모를 활용하고, 기존의 로직을 최대한 변경하지 않는 선에서 리디자인을 진행한다.

해결해야 할 문제점

2013년 처음 론칭한 모바일 K-town 서비스는 웹 중심의 유지보수로 인해, 모바일 사용자의 니즈를 해결 할 핵심 기능의 업데이트는 보류되고 문제점은 내부 인력의 대응으로 운영되어 왔다. 이로 인해 운영비용이 높게 발생하고 사용자들의 불편함은 지속적으로 누적되어 모바일 사용자가 감소하였다.

  • 네이티브 앱의 장점을 활용하지 못한 서비스

  • 명확하지 않은 사용자 역할로 목적이 희미해진 UX 구조

  • 낮은 심미성과 불명확한 브랜딩으로 낮은 매력도와 웹과의 일관성

  • 혼란스러운 UX Writing과 반복되는 기능 배치 등의 잘못된 설계

  • 관리자에게 의존하여 운영되는 것이 전제가 되어있는 설계

목표

  • 네이티브의 장점을 적극적으로 활용하여 모바일 최적화 서비스를 제공

  • 사용자가 주체적으로 활용할 수 있는 프로덕트를 제공하여 사내 관리자의 업무를 최소화

  • 산발적인 업데이트에도 일관성을 유지할 수 있는 디자인 시스템 설계

  • 브랜드 아이덴티티를 명료히하여 일관성과 심미성 전달

  • 각 사용자가 목적을 도달하는데에 필요한 기능과 만족스러운 경험 제공

제안

사용자가 목적을 도달하는 동안에 발생하는 문제점을 줄이기 위해 니즈에 따라 분기된 역할을 명료히 한 서비스로 재설계한다. 기존 서비스의 혼란스러운 UX와 심미성이 낮은 UI를 개선하여 서비스의 매력을 높히고, 지속되는 업데이트에도 일관성이 유지될 수 있도록 디자인 시스템을 공유한다.

이번 리디자인에서의 가장 중요한 부분은 니즈에 따라 사용자를 정확하게 정의하고, 그들이 목적을 수행하는 기능을 추가하며 혼란을 최소화하는 것이다. 이는 사용자가 K-town의 사용을 습관화할 수 있도록 도울 것이다.

디자인 프로세스

프로젝트 팀과 나의 역할

iOS와 Android 모바일 개발자 2명, 백엔드 개발자 1명 그리고 나 이렇게 4명으로 구성되었다. 나는 문제점 파악과 최종 디자인까지 디자인 전반을 담당하였다. 프로젝트 진행과정에서 Excutive 멤버와 시니어 디자이너가 주기적으로 서비스의 진행방향을 점검하였다.

프로덕트가 달성한 것

  • 프로덕트 End to End 리디자인과 브랜드 리디자인

  • 문제점 파악 그리고 해결방안과 개선안 도출

  • 빌드를 위한 자료 수집 및 UX Research

  • 프로덕트의 일관성을 위한 시각 시스템 업데이트

  • 기존 데이터구조 안에서 문제점 해결을 위한 서비스 재설계

Empathise

리서치

서비스의 목적과 구성를 파악하고, 존재하는 문제점을 파악하기 위해 아래와 같은 방법을 사용하였다.

기존 프로덕트 스터디

기존의 웹과 앱서비스를 분석하여 서비스를 학습하고, Frictions이 발생하는 지점을 확인한다.

1. 사이트 맵

서비스의 모든 페이지를 확인하며 사이트맵을 작성하고, 사용시 불편한 점과 문제가 될만한 UX들을 체크한다.

2. 모바일앱 사용자경험 분석

서비스의 문제점들을 발견하기 위해 페이지 depth 단위로 상세히 기재하여 리스트화한다. 여기서 사용자가 느낄 예측 가능한 문제점들도 함께 작성한다. 이 문서는 추후 아래의 Ideate 단계에서 추가적으로 활용되었다.

가설

나는 위의 과정을 통해 서비스가 해결하고자 했던 원래의 니즈를 찾고, 그 과정에서 사용자가 서비스를 사용하며 마주한 불만을 초기 가설로 수립함으로써 문제를 정확히 정의하고자 하였다.

사용자 니즈

  • 일반 사용자 : 방문 예정인 한인 업소에 대한 정보를 미리 확인하고 얻을 수 있는 할인 혜택이 있는지도 알고 싶어

  • 일반 사용자 : 다른 지역에 방문할 일이 있는데 괜찮은 곳이 있는지, 특별한 이벤트나 할인혜택이 있는지 알고 싶어

  • 업소 등록 사용자 : 나의 업소를 홍보해서 더 많은 사람을 방문하도록 하고 싶어

사용자 불만

  • 검색 종류가 너무 많아서 어떻게 해야 내가 원하는 정보를 찾을 수 있을지 모르겠어

  • 특정 업소가 아닌 새로운 업소나 근처의 괜찮은 업소들에 대한 정보 어디서 볼 수 있는건지 모르겠어

  • 업소 등록은 쉽게 했는데, 정보 수정을 어떻게 해야할 지 모르겠어

인터뷰

1. 사용자 인터뷰

우리는 20대에서 40대의 포텐셜 유저 5명, 기존 제공 방식을 경험한 적이 있는 5명을 인터뷰하였다.

🎤  제너럴 인터뷰

  • 사용자 그룹의 동기, 유입경로, 목적, 주로 사용한 장소와 이유

  • 포텐셜 사용자 그룹의 자유로운 사용 후 서비스의 목적과 인상

  • 서비스 사용에 대한 장점과 단점

👀  사용 방식 관찰

  • 이름을 모르지만 카테고리를 아는, 이름과 카테고리를 아는, 이름만으로 업소 정보 검색하기

  • 리뷰 작성하기, 리뷰찾기

  • 이벤트·쿠폰 검색하기

✨  니즈 인터뷰

  • 사용자의 요구사항 수집

  • 목적 도달과정 내에서 불편했던 지점 수집

  • 개선 어트렉션 포인트 조사

2. 제휴자 인터뷰

광고 제휴자의 업소를 직접 방문하여 관찰 혹은 인터뷰를 진행하거나 전화 인터뷰를 진행하였다.

🎤  만족도 인터뷰

  • 동기, 유입경로, 목적, 주로 사용한 장소와 이유

  • 서비스 사용에 대한 장·단점

👀  사용방식 관찰

  • 활용 및 정보 관리 방식 관찰

  • 일반사용자들의 쿠폰 및 이벤트 활용 인터뷰

✨  니즈 인터뷰

  • 제휴자의 요구사항 수집

  • 목적 도달과정 내에서 불편했던 지점 수집

  • 개선 어트렉션 포인트 조사

인터뷰 과정에서 우리는 아래와 같은 새로운 인사이트를 얻을 수 있었다:

  • 나의 업소가 메인에 노출되고, 비용을 지불하더라도 자주 노출되었으면 좋겠다.

  • 쿠폰 정보를 올려두어도 쿠폰 활용이 불가한 업소가 있다.

  • 업소를 한 곳만 운영하기보다는 지점 형태 혹은 별도의 업소 두 곳 이상을 운영하는 경우가 존재한다.

  • 결제 주체가 아닌 업소의 매니저가 마케팅을 담당하는 경우, 결제 주체의 등록 계정을 공유하여 관리하게 된다.

Define

위와 같은 조사과정을 통해 얻은 인사이트들을 기반으로 가설을 검증하였다. 또한 리서치를 통해 얻은 인사이트를 기반으로 새로운 니즈를 발견하고 추가하였다.

✅  일반 사용자 : 다른 지역에 방문할 일이 있는데 괜찮은 곳이 있는지, 특별한 이벤트나 할인혜택이 있는지 알고 싶어

문제점

웹 중심의 산발적인 업데이트를 통해 약화된 모바일 사용자 경험을 강화하고, 시간이 지남에 따라 희미해진 브랜드 경험을 개선한다. 더불어, 내부 관리자에게 의존하지 않고 사용자가 주체적으로 서비스를 활용할 수 있도록 한다.

  • 네이티브 앱의 장점을 활용하지 못한 서비스

  • 명확하지 않은 사용자 역할로 목적이 희미해진 UX 구조

  • 낮은 심미성과 불명확한 브랜딩으로 낮은 매력도와 웹과의 일관성

  • 혼란스러운 UX Writing과 반복되는 기능 배치 등의 잘못된 설계

  • 관리자에게 의존하여 운영하는 것이 전제가 되어있는 설계

타깃과 페르소나

서비스의 주 사용자는 유학생 학부모, 자영업자, 직장인, 유학생순이며, 10대 후반에서 50대 중반까지로 다양한 연령대가 사용하나 35-45세 사용자가 가장 많다. 남·여의 사용 비율은 차이가 크지 않았으나, 여성 사용자 수치가 좀 더 높았다. 리서치 과정과 프로덕트 타깃의 범위 그리고 정의된 문제점을 바탕으로 해결안 수립의 모델이 되는 퍼르소나를 정의했다.

디자인 원칙

기존 서비스의 문제점들을 확인하는 과정에서 리디자인의 주요 원칙을 설정하였다. 프로덕트 자체에 대한 이야기이기도 하지만, 개발 및 디자인을 하는 과정에서도 적용될 수 있는 원칙을 수립하고 팀원들과 공유한다.

Seamless integration

목적이 다른 사용자들이 섞여있는 프로덕트가 사용자에게 혼란스럽지 않도록 설계하고, 각각의 목적도달 과정의 마찰을 최소화한다.

Simple and Clear

개발 결과와 과정을 심플하고 명료하게 진행하여 우선순위, 서비스 위계 같은 큰 핵심요소부터 섬세한 요소까지 효율적으로 디자인한다.

Follow the design system

담당 팀이 자주 변경되는 사내환경에서 인수인계가 쉽도록 고려하고, 사용자에게 브랜드와 사용 경험을 일관성 있게 제공하도록 설계한다.

개발 범위

웹 서비스의 구조를 기반으로 모바일 경험의 최적화를 기본 개발범위로 지정한다. 다만, 웹의 구조에도 영향이 가는 사항이라도 중요도에 따라 개발 범위에 포함될 수 있도록 논의하여 개발 범위를 지정하였다.

해당 스터디케이스에 작성된 범위의 프로젝트 진행 과정 중 추가 요청되는 기능들이 발생한 경우 요청 기능의 우선순위를 파악하여 범위에 추가하였다. 개발 범위 미포함으로 분류된 내용도 요구사항 정의서에 아카이빙하여 추후 개발을 이어나갈 수 있도록 하였다.

Ideate

아이디어 도출

마인드 맵

마인드맵을 통해 정의된 문제점을 중심으로 아이디어를 자유롭게 확장해 나간다. 개발 리소스와 개발 방식에 제약 없이 최대한 다양한 솔루션과 문제점 해결에 필요한 기능을 작성하는 것에 중점을 두고 진행한다.

아이디어 분류와 고도화

마인드맵에서 문제 해결에 활용할만한 아이디어들을 그룹화하고 상세한 기능으로 발전시킬 수 있도록 아래와 같이 확장하고 정의하여 최종 개발 여부를 논의한다.

사용자가 어떻게 더 쉽게 서비스를 사용할 수 있을까?

  • 사용자 역할 분기 : 서비스를 사용하는 사용자들의 주목적에 맞추어 두 가지의 사용자로 역할을 분기한다.

  • 사용성 개선 : 카메라, 라이브러리, 전화, GPS, 도움이 되는 다른 앱으로의 쉬운 전환 등 네이티브 앱의 특장점을 사용자의 최종 목적 도달에 최대한 활용한다.

어떻게 명확한 정보에 빠르게 도달하게 만들 수 있을까?

  • 잘못된 UX의 개선 : 통합되지 않아 혼란을 가중하던 검색 경험을 통합 개선하여 어떤 형태의 검색이든 사용자의 고민을 최소화하는 검색 경험을 제공한다.

  • 정보 위계 구조 디자인 : 스토어의 상세 정보들을 사용자의 필요 순위에 맞게 재배치하여 사용자가 적정한 속도로 정보를 인지하고 필요에 따라 부수적인 정보를 확인하여 정보를 소화할 수 있도록 설계한다.

업소 사용자들에게 어떻게 매력적인 프로덕트가 될 수 있을까?

  • 업소 정보 CRUD : 업소 정보를 직접 등록하고 관리하며 내부에서는 승인하여 리얼서버에 노출하는 구조로 변경하여, 내부 인력의 업무 과부하를 줄이고 사용자들에게 더 정확한 정보를 빠르게 제공한다.

  • 매니저 임명 기능 : 업소 등록 사용자들의 운영 방식을 커버할 수 있도록 업소 관리 사용자는 어드민과 매니저 두 가지로 분기되도록 하며, 두 역할은 결제 가능여부로 차이를 둔다.

  • 예약 CTA : 일반 사용자에겐 서비스를 통한 예약 시 혜택을 제공하고, 업소 사용자에겐 추가적인 예약발생이 가능하도록 설계한다. 해당 기능에 따라 예약 기능, 시간 변경 요청, 채팅 등의 추가적인 기능을 개발한다.

  • 통계 기능 : 업소 정보에 도달하는 사용자에 관한 정보를 제공한다. 업소사용자의 필요에 따라 선택적으로 확인할 수 있도록 지불하는 플랜에 따라 정보 도달 범위를 달리한다. 이는 프로모션 플랜에 대한 동기를 부여한다.

어떻게 일관되고 아름다운 프로덕트를 제공할 수 있을까?

  • 브랜드 디자인 개선 : 회사 메인서비스의 디자인 아이덴티티와 일관성을 갖도록 서비스의 아이덴티티를 개선한다.

  • 프로덕트 디자인 시스템 설계 : 팀이 자주 변경되는 회사 특성에 대비하여 일관성이 흐트러지지 않도록 프로덕트의 시스템을 설계하고 공통 UI에 대한 가이드를 제공한다.

  • 코드 기반의 디자인 : 언어변환 대응으로의 확장과 앱 용량 이슈와 해상도에 효율적으로 대응하기 위해 이미지형 버튼이 아닌 코드기반의 디자인을 설계한다.

요구사항 정의

최종 합의가 도출된 기능들을 요구사항 정의서로 정리한다. 기능은 사용자 역할을 기준으로 작성되며 상세한 카테고리와 개발내용, 개념 등을 개괄하여 작성한다. 더불어 우선순위에 따라 분기된 개발 범위도 확인할 수 있도록 작성하여 개발팀과의 커뮤니케이션에 활용한다.

프로덕트 상세

01

다른 목적을 갖고 있는 각각의 사용자들,
어떻게 혼란을 줄이고 사용자의 목적을 쉽게 달성할까요?

대화형 온보딩

기존 앱 소개형 온보딩에서 사용자의 목적에 맞는 페이지로의 연결을 도와주는 대화형 온보딩으로 변경하여 사용자를 가이드한다.

사용자 맞춤 메뉴 구성

업소 검색과 업소 등록의 두가지 목적이 섞여 혼란스럽던 메뉴에서 각 사용자의 니즈에 맞게 분리된 메뉴를 제공해 사용자의 혼란을 최소화시킨다.

다양한 니즈를 위한 역할 변경

사용자의 필요에 따라 쉽게 역할을 변경할 수 있도록 설계하여, 다양한 니즈를 해결할 수 있는 입체적인 프로덕트를 제공한다.

02

한인 업소 검색,
어떻게 명확한 정보에 빠르게 도달할 수 있을까?

혼란을 줄이는 강력한 통합 검색

기존 프로덕트에서 불필요하게 중복되어 혼란을 가중하는 검색들을 하나로 통합하여 제공한다. 일반 사용자의 탐색과 제휴 사용자의 업소 노출을 확장시킬 수 있도록 메인에서 업소 정보가 드러나게 설계한다.

그룹화된 정보와 니즈에 맞는 CTA

사용자가 적절한 속도로 정보를 인지하고 필요에 따라 부수적인 정보를 확인하여 정보를 소화할 수 있도록 시각요소와 정보 구조를 재설계한다. 더불어, 사용자들의 니즈를 충족시킬 CTA를 구성한다.

03

한인 업소 홍보,
어떻게 쉽고 효율적으로 운영할 수 있을까요?

목적에 도달하는 직관적인 UX

업소정보 수정 및 삭제가 불가능하여 서비스 운영에 큰 내부 인력 활용과 CS를 발생시키던 기존 서비스구조에서, 업소등록 사용자들이 언제든 쉽게 관리할 수 있는 프로덕트로 개선한다. 사용자들의 니즈에 맞춰 매니저 임명 등의 새로운 기능을 추가하여 효율적인 프로덕트가 되도록 설계한다.

쉽고 간편한 예약관리

새로운 예약을 실시간으로 확인하고 수락할 수 있도록 설계하고 언제든 예약일정과 지난일정들을 확인할 수 있다. 예약시간 변경요청과 채팅 등의 기능을 통해 사용자가 직면할 수 있는 이슈해결이 쉽고 간편하도록 설계한다.

스마트한 관리

효율적인 관리방식을 찾을 수 있도록 통계 기능을 제공한다. 플랜에 따라 자료 열람의 범위를 다르게 설계하여 프로모션 플랜에 대한 동기부여를 제공한다.

Evaluation

평가

프로젝트 론칭 이후 팀 전원이 회사 내규에 따라 다른 프로젝트로 배정되며 정성, 정량적인 평가는 진행되지 않았다. 따라서, 나는 아래와 같은 세가지 방법으로 프로젝트 평가를 진행하였다.

평가 방식

1) 주기적으로 진행되는 내부 평가와 론칭에 대한 내부 평가

2) 블랙박스·화이트박스 기능 테스트

3) 초기 설정한 프로덕트의 목표와 디자인 원칙을 기준으로 도달·준수 여부 평가

세번째 평가 항목은 최종 론칭 리뷰에 참여한 회사 멤버들을 통해 평가되었다:

프로젝트 초기 수립 목표

  • 네이티브의 장점을 적극적으로 활용하여 모바일 최적화 서비스를 제공

    • 🧐   사용자가 주체적으로 활용할 수 있는 프로덕트를 제공하여 사내 관리자의 업무를 최소화

    ➡️  설계상 기존 서비스에 비해 업무로드를 줄일 수 있으나 실제 효용성 관찰 필요

  • 산발적인 업데이트에도 일관성을 유지할 수 있는 디자인 시스템 설계

  • 브랜드 아이덴티티를 명료히하여 일관성과 심미성 전달

  • 각 사용자가 목적을 도달하는데에 필요한 기능과 만족스러운 경험 제공

자가 평가

프로젝트의 아쉬운 점

평가 부분에서 언급했듯 프로젝트 완료 후, 프로젝트의 실질적인 운용을 돌아볼 수 없이 바로 다른 프로젝트에 투입되었던 점이 아쉽다. 일정 시기가 지나 내부에선 사용자가 늘었고, 만족스럽다는 평을 얻었으나 상세한 내역을 확인할 수 없었다.

프로젝트 이후 한달 정도의 관찰 기간을 확보하고, 유의미한 성과가 있음을 확인하는 것 또한 프로젝트의 일정이라고 설득할 수 있는 기회가 있었다면 더욱 오래 유지되는 프로덕트로 남지 않았을까 하는 아쉬움이 있다.

이번 프로젝트를 통해 배운 것

  • 고객 직접 방문을 통해 활용사례 관찰·인터뷰하는 것

  • 기존 서비스의 데이터 구조 안에서 개선사항을 찾는 것

  • 프로덕트 디자이너로서 프로젝트를 운영하는 것

다음 프로젝트에서 더욱 잘할 수 있는 것

  • 유저 인터뷰와 고객 방문관찰을 통한 유저리서치

  • 기존 구조를 토대로 end to end의 재설계

  • 이슈상황에서 관계자들을 설득하는 소프트스킬
